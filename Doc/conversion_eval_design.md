# 前処理ツール比較評価 - 全体設計ドキュメント

**作成日**: 2026-04-17
**最終更新**: 2026-04-25
**対象**: YPS/COBOL 知識継承プロジェクト WS1(前処理ツール比較評価)
**位置付け**: **PoC(概念実証)段階**。手元 539 ファイルで各パターンを比較し、本番 12 万ファイル展開への採用判断の根拠を得る(運用前提は第 13 章参照)
**目的**: レガシーシステムの各種ドキュメントを Markdown 化するにあたり、複数ツール・複数経路を比較し、採用判断の根拠となるデータを得る

## 1. 背景と目的

### 1.1 なぜ比較評価が必要か

対象ドキュメント群は以下の特徴を持ち、単一ツールで全てを処理する方針が妥当か判断できない:

- 形式が多岐にわたる(.doc / .docx / .xls / .xlsx / .pdf / .rtf / .txt 等)
- 旧形式(.doc / .xls / .rtf)の比率が高い(件数ベースで約 30%)
- チームごとにフォーマット思想が異なる
- 本番規模は 12 万ファイル・1.2GB(設計書のみ)。精度だけでなく**処理速度**も重要

### 1.2 評価の成果物

1. **各パターンの変換結果**(Markdown ファイル)
2. **評価レポート**(CSV + サマリー)。人の目評価と機械評価を統合
3. **採用判断の根拠資料**(報告用)

### 1.3 スコープの境界

**本評価に含むもの**:
- Markdown 化までの前処理パイプライン
- 速度・品質・ノイズ・失敗率の測定

**本評価に含まないもの**:
- GraphRAG への投入処理(採用決定後、別タスク)
- Dify ワークフロー側の設計(WS2 の範疇)
- ファイル関係抽出(WS3 の範疇)

**次工程(RAG)との関係**:
本評価で出力された Markdown は、そのまま RAG(既存 GraphRAG / Dify ベクトル RAG 等)への投入対象になる。つまり**本評価の出力は単独の成果物ではなく、次工程の入力でもある**。

ただし「RAG に投入したときの回答品質」の定量評価は WS2 の範疇とし、WS1 では扱わない。WS1 ではあくまで前処理の品質(Markdown としての読みやすさ、構造保持、ノイズ量)に集中する。

この切り分けの理由:
- 前処理と RAG 両方の品質を同時に評価すると変数が爆発し、「どこが問題か」の切り分けが不能になる
- 段階的に変数を固定(まず前処理を決める → その後 RAG 側を評価)する方が、評価設計として健全
- ただし「RAG に投入できる最低限の品質か」という観点は、目視評価の任意項目として残す(第 4.4 節)

### 1.4 第 1 段階のポリシー

本ドキュメントは段階的な評価を前提にしている。**第 1 段階**では以下のスタンスを取る:

- **ノイズ除去は行わない**。各ツールの**素の出力**をそのまま評価する
  - 理由: ツール本来の品質を直接比較するため。正規化のバイアスを排除し、「どのツールがノイズを出しやすいか」も評価情報として残す
- **ノイズ除去のコードとルール定義は用意する**が、設定ファイルで無効化しておく
  - 将来(第 2 段階以降)、設定変更だけで除去を有効化できるよう、実装の受け皿は先に作る
  - 記述や設計意図を残すことで、「なぜこの形にしたか」が引き継ぎ時にも辿れる
- **ノイズ系の指標は計測する**。除去はしないが、計測値は評価軸として使う

この方針は第 4.3 節(評価指標)と第 8 章(ノイズ除去の設計)で具体化する。

### 1.5 実行環境: Windows で完結

本評価は **Windows 環境で完結**させる。既存の WSL / Docker / GraphRAG とは**完全に独立した実験プロジェクト**として扱う。

**理由**:
- Office COM が Windows でしか動作しないため、全パターンを同条件で比較するには Windows が最適
- 既存 GraphRAG との統合は「採用ツールが決まった後」の別タスク。評価フェーズではパイプライン統合は不要
- WSL / Docker / ネットワーク境界をまたがない構成の方がシンプルで、比較実験の再現性が高い
- 引き継ぎ先の業務担当者も Windows 環境であるため、最終的な運用環境に近い

**使用環境**:
- Windows 本体
- Python 3.11+
- Microsoft Office(COM 使用のため)
- LibreOffice(LibreOffice 経由の前処理に使用、**利用可否は社内ポリシー確認中**)

**配置先**: 任意のローカルフォルダ(例: `C:\project\conversion_eval\`)

**採用決定後の組み込み**: 採用ツールが決まった後、WSL 側の既存 `graphrag-api` に変換機能として組み込む作業は別タスクとして実施。採用ツールが COM の場合は Windows 連携(小型 FastAPI または定期実行スクリプト)を追加設計する。

## 2. 対象ファイル形式と規模感

### 2.1 試し範囲(現在手元にある 539 ファイル / 79.7MB)

| 拡張子 | 件数 | 容量 | 変換難易度 | 備考 |
|---|---|---|---|---|
| .doc | 129 | 15.9MB | **高**(旧形式) | 件数ベース 24% |
| .docx | 189 | 6.6MB | 低 | 件数最多 |
| .xls | 32 | 10.1MB | **高**(旧形式) | |
| .xlsx | 30 | 1.2MB | 低 | |
| .pdf | 108 | 24.5MB | 中 | スキャン PDF の可能性あり |
| .rtf | 25 | 102.2MB | **高** | 1 ファイル平均 4MB と巨大 |
| .txt | 14 | 10.9MB | 極低 | |
| .bik / .bpg / .bca / .bci | 4 | 197KB | 不明 | YPS/COBOL 独自形式。**試験的にパイプラインに流して挙動を観察**(第 2.5 節) |

### 2.2 本番規模(推定)

- 設計書(DD 書)のみで **12 万ファイル・1.2GB**
- ソースコード **1GB**
- 試し範囲は全体の約 0.5%

### 2.3 本評価の対象外とするファイル種別

以下は本 WS1(前処理ツール比較評価)の対象外とする:

- **COBOL ソース(.cbl / .cob / .cpy)**
  - 理由: COBOL ソースは Markdown 変換しても意味がない。テキストとしてはそのまま読めるし、必要なのは CALL/COPY などの関係抽出
  - **WS3(ファイル関係抽出)の範疇**。そちらで別設計で扱う
- **バグレス(BAGLES)関連の独自ファイル以外のソース系**
  - YPS が生成するソース系ファイル全般(.cbl / .cob / .cpy を含む)は、設計書類とは完全に分離して扱う

本 WS1 は、**設計書・ドキュメント系のファイルを Markdown 化する**ためのツール比較に集中する。ソースコード系ファイルは混ぜない。

### 2.4 規模感から導かれる評価観点

- **速度**: 12 万ファイルへの外挿時間が現実的か
- **失敗率**: 10% 失敗しても 1.2 万ファイルが漏れる。失敗時の代替手段が必須
- **並列化可能性**: 単発実行時間を並列化でどこまで短縮できるか

### 2.5 独自形式(.bik / .bpg / .bca / .bci)の扱い

本件では YPS/COBOL の富士通系開発環境に由来する独自バイナリ形式が数件混在している。これらは既存ライブラリでは想定外の入力だが、**試験的にパイプラインに流して挙動を観察**する。

**試す狙い**:
- MarkItDown / Docling が**どう振る舞うか**を知る(エラーで落ちるか、無言で変な出力を出すか)
- 本番 12 万ファイルに独自形式が混入した場合の運用品質(エラーハンドリングの質)を評価する
- 意外にもテキストが取り出せる可能性がゼロではない

**試さないこと(今回のスコープ外)**:
- 独自形式のネイティブパース(富士通系ツールでのエクスポート、独自パーサの実装など)
- 形式の詳細調査(業務担当者への質問、マジックナンバー解析など)

**実装上の扱い**:
- 他形式と同じパイプラインに流す
- 前処理・本変換の成功/失敗、エラーメッセージをレポートに記録する
- 成功した場合は出力 Markdown を保存し、目視で確認可能にする
- 第 1 段階の結果を見て、次の対応(調査を深めるか、完全対象外にするか)を判断する

**期待される結果の仮説**:
- ほぼ全パターンで前処理か本変換のどちらかが失敗する可能性が高い
- 失敗が「明確なエラー」か「サイレントな異常出力」かで、ツール間の差が見える可能性
- 失敗の仕方そのものが比較評価の材料になる

## 3. 比較パターンの整理

### 3.1 変換経路の 2 次元マトリクス

変換経路は「**前処理**」と「**本変換**」の 2 段階で整理できる。

#### 3.1.1 前処理(旧形式を新形式に揃える / PDF を Word 形式に変換する)

| 前処理方式 | 用途 | 実行環境 |
|---|---|---|
| なし(直接) | 新形式(.docx / .xlsx / .pdf 等)をそのまま扱う | 任意 |
| LibreOffice 経由 | 旧形式を新形式に変換、PDF を Draw 経由で `.docx` 化 | Windows / WSL どちらも可(**利用可否は社内ポリシー確認中**) |
| Office COM 経由 | Word/Excel 本体で旧形式や PDF を開き、新形式に保存 | **Windows 必須**(Office インストール必須) |

**注**: LibreOffice は独立したアプリケーション(オフィススイート)であり、Python ライブラリではない。利用には Windows にインストールするか、ポータブル版を配置する必要がある。本書ではインストール可能であることを前提に記述しているが、**社内ポリシーによっては利用不可の可能性があり、その場合はパターン C, D を実施対象から外し、A, B, E, F, G の 5 パターンで評価する**(第 3.5 節参照)。

**PDF の扱い**: Word は 2013 以降「PDF Reflow」機能で PDF を編集可能な `.docx` に変換できる。COM 経由(パターン A/B/G)では Word で PDF を開き `.docx` 化した後に本変換を行う。LibreOffice は PDF を Draw で開き `.docx` エクスポートする経路を取る。詳細は第 3.6 節「PDF 処理の特殊事項」参照。

#### 3.1.2 本変換(Markdown 化)

| 変換ツール | 強み | 弱み |
|---|---|---|
| MarkItDown(Microsoft) | 軽量、多形式対応、高速 | 複雑な表や図の扱いが雑 |
| Docling(IBM) | レイアウト解析が強力、表構造を正確に保持 | やや重い、GPU 推奨 |
| **Office COM 直接 Markdown 化** | Word/Excel が解釈した結果を取れる(フィールド評価済み、目次展開、独自スタイル認識) | **実装コスト大**。段落・表・見出しを API で読み自前で Markdown 組み立て |

#### 3.1.3 マトリクス(組み合わせ表)

```
                │ MarkItDown       │  Docling         │  COM 直接 Markdown │
────────────────┼──────────────────┼──────────────────┼────────────────────┤
直接変換        │  パターン E      │  パターン F      │  (N/A)             │
(前処理なし)   │                  │                  │                    │
────────────────┼──────────────────┼──────────────────┼────────────────────┤
LibreOffice     │  パターン C      │  パターン D      │  (N/A)             │
前処理あり      │                  │                  │                    │
────────────────┼──────────────────┼──────────────────┼────────────────────┤
Office COM      │  パターン A      │  パターン B      │  パターン G        │
前処理あり      │                  │                  │  (前処理と本変換を  │
                │                  │                  │   COM 内で一体実行) │
```

**合計 7 パターン**が比較対象となる。パターン G は「COM で開いて、そのまま COM API 経由で構造を読んで Markdown を組み立てる」一体型で、前処理と本変換の工程が分離されない。

**PDF の扱い**: PDF は **7 パターン全てで処理対象**とする(第 3.6 節参照)。これにより「Word/LibreOffice の PDF Reflow が下流ツールの Markdown 品質を改善するか」という比較軸が得られる。

各パターンの PDF 処理フロー:

| パターン | PDF 処理フロー |
|---|---|
| A | Word が PDF を開く → `.docx` で保存 → MarkItDown |
| B | Word が PDF を開く → `.docx` で保存 → Docling |
| C | LibreOffice Draw が PDF を開く → `.docx` エクスポート → MarkItDown |
| D | LibreOffice Draw が PDF を開く → `.docx` エクスポート → Docling |
| E | PDF → MarkItDown 直接 |
| F | PDF → Docling 直接 |
| G | Word が PDF を開く → COM API で構造抽出 → Markdown 組み立て |

### 3.2 パターン間の比較意義

各パターンの比較で見えるもの:

| 比較 | 分かること |
|---|---|
| A vs C(MarkItDown × COM vs LibreOffice) | **旧形式前処理の品質差**(COM の優位性があるか) |
| B vs D(Docling × COM vs LibreOffice) | 同上(Docling で検証) |
| A vs B(COM × MarkItDown vs Docling) | **COM 経由時の本変換ツール差** |
| C vs D(LibreOffice × MarkItDown vs Docling) | **LibreOffice 経由時の本変換ツール差** |
| E vs F(直接 × MarkItDown vs Docling) | **新形式の本変換ツール差**(前処理の影響を受けない純粋比較) |
| A+C vs B+D | **前処理差 vs ツール差**、どちらがより効くか |
| G vs A/B | **COM 一体型 vs COM + 既製ライブラリ**、Word/Excel が解釈した情報(フィールド評価済み、目次、独自スタイル)を直接使うことで、MarkItDown/Docling が拾えない情報が取れるか |
| G vs 全体 | **Office ネイティブ変換の上限品質**。COM が最高品質を出せるなら、その差がどれくらいあるかで他ツールの相対評価ができる |

### 3.3 パターン G(COM 直接 Markdown 化)の位置づけ

**狙い**: Word/Excel 本体が解釈した結果(目次の評価済み文字列、フィールド展開結果、独自スタイルの認識、結合セルの扱い)を、MarkItDown や Docling が拾えない場合の**品質上限**として測定する。

**実装上の注意**:
- COM API で段落・表・見出しを順に読み、自前の Markdown 組み立てロジックが必要(数百行規模)
- 実装コストは他パターンより明らかに大きい。**第 1 段階では簡易実装で十分**とし、本格的な Markdown 整形は評価結果を見てから判断
- 簡易実装版でも、「COM だから取れる情報」の有無を確認できれば評価の役に立つ

**簡易実装の最小要件**:
- 見出しスタイルを認識し `#` `##` に変換
- 段落を空行区切りで出力
- 表は Markdown テーブル形式で出力(崩れても可、段階的に改善)
- 画像は `![画像]()` プレースホルダで出力
- フィールドは展開済み文字列として出力(COM の特権機能)

### 3.4 評価対象外の組み合わせ

- **LibreOffice 直接 Markdown 化**: LibreOffice には Markdown エクスポート機能がなく、同等の処理を自前で書くと COM 版とほぼ同じ実装コストがかかる。かつ Office ネイティブ機能(フィールド等)の展開もできないため、追加価値が薄い
- **python-docx / openpyxl による自前 Markdown 化**: MarkItDown の内部実装と本質的に同じ。独自に書く意義が薄いため今回は対象外
- **独自形式(.bik / .bpg 等)**: 既存ライブラリでは正規対応不可。**試験的にパイプラインに流す(第 2.5 節)**
- **Markdown の自然言語化(verbalization / narrative transformation)**: 表やリストを LLM で散文に変換する後処理は、WS1 のスコープ外とする(理由は下記)

**注**: PDF は当初「直接変換のみ(パターン E, F)」とする方針も検討したが、Word/LibreOffice の PDF Reflow が下流ツールの Markdown 品質に与える影響を測定する価値があるため、**全 7 パターンで処理する判断に変更**(第 3.6 節)。

#### Markdown の自然言語化を WS1 から外す理由

「Markdown 化した後、表やリストを LLM で自然言語の散文に変換すると RAG 精度が上がる」というアプローチが知られている(verbalization、narrative transformation、document expansion 等の名称)。本評価では以下の理由で **WS1 のスコープ外**とする:

- **前処理品質の比較が混濁する**: 自然言語化を挟むと「ツール本来の Markdown 品質」と「LLM 自然言語化の効果」が切り分け不能になる。第 1.4 節「ノイズ除去をしない」と同じ思想
- **評価コストの爆発**: 7 パターン × 自然言語化(あり/なし)= 14 パターン、PoC スコープを超える
- **LLM 依存が入る**: ローカル LLM(Gemma 4 等)の性能・プロンプト・温度が結果に影響、再現性が下がる。「MarkItDown の品質を見たいのに Gemma 4 の性能を見ることになる」という問題
- **本質的に RAG 投入時の問題**: 自然言語化は RAG の前段の augmentation であり、RAG 手法と組み合わせて評価するのが自然 → **WS2 で扱うべき**

WS2 では「同一 Markdown × 3 RAG 手法 × 投入形式(直接 / 自然言語化後)」のように評価軸を拡張する余地がある(`rag_evaluation_design.md` 参照)。自然言語化の効果は質問種別との相性も大きい(事実検索では効きやすい、構造を辿る質問では逆効果の可能性)ため、WS2 の質問カテゴリ別評価と相性が良い。

### 3.5 LibreOffice が利用不可の場合の退避プラン

社内ポリシーにより LibreOffice が利用できないと判明した場合、以下の方針でパターンを縮退させる:

**縮退案**: 5 パターン構成(A, B, E, F, G のみ)

```
                │ MarkItDown       │  Docling         │  COM 直接 Markdown │
────────────────┼──────────────────┼──────────────────┼────────────────────┤
直接変換        │  パターン E      │  パターン F      │  (N/A)             │
(新形式のみ)   │                  │                  │                    │
────────────────┼──────────────────┼──────────────────┼────────────────────┤
Office COM      │  パターン A      │  パターン B      │  パターン G        │
前処理あり      │                  │                  │                    │
```

**縮退による影響**:
- 旧形式(.doc / .xls / .rtf)の変換は **COM 経由のみ**になる
- 「LibreOffice と COM の品質・保守性比較」ができなくなる → COM 依存度の評価が純粋にはできない
- 代わりに、COM が使える環境なら最も高品質な変換が期待できる(トレードオフとして許容範囲)

**縮退の切り替え方法**:
- 設定ファイル `config/patterns.yaml` で パターン C, D の定義をコメントアウトするだけ
- 実行時に `--patterns a,b,e,f,g` と指定するだけ
- コード変更は不要

LibreOffice の利用可否は第 14 章「未解決事項」に記載。確認完了後に本節の記述を更新する。

### 3.6 PDF 処理の特殊事項

PDF を全 7 パターンで処理する判断(第 3.1.3 節)に伴い、PDF 固有の挙動と運用上の注意を整理する。

#### 3.6.1 Word PDF Reflow の特性

Word は 2013 以降、PDF を編集可能な `.docx` に変換する「PDF Reflow」機能を持つ。COM 経由(パターン A/B/G)はこの機能を利用する。

```python
# 概念コード(パターン A/B/G の前処理部分)
import win32com.client
word = win32com.client.Dispatch("Word.Application")
word.Visible = False
doc = word.Documents.Open(r"C:\path\to\sample.pdf")
doc.SaveAs2(r"C:\path\to\sample.docx", FileFormat=16)  # 16 = wdFormatXMLDocument
doc.Close()
```

**重要な制約**:

- **OCR 機能ではない**。PDF 内に埋め込まれたテキスト情報を抽出して再構成する仕組み。**スキャン PDF(画像のみ)からはテキストを取得できない**(画像が貼り付いた `.docx` になるだけ)
- **複雑なレイアウト(複数段組、結合セルが多い表、数式)は崩れやすい**
- **処理時間が他形式より大幅に長い**(ページ数とレイアウト複雑度依存。1 ファイル数十秒〜数分のケースあり)

#### 3.6.2 ダイアログ抑制

Word で PDF を初回に開くと「Word は PDF を編集可能な Word 文書に変換します」というモーダルダイアログが出て処理が止まる。COM 経由実行時は事前に以下のいずれかで抑制する必要がある:

- レジストリ設定: `HKEY_CURRENT_USER\Software\Microsoft\Office\<version>\Word\Options\DisablePDFReflowWarning = 1`
- COM 側: `Application.DisplayAlerts = wdAlertsNone`(ただしダイアログ種別によっては効かない場合あり)

**実装上**: パターン A/B/G の COM preprocessor 起動時に上記設定を行うラッパーを用意する。

#### 3.6.3 LibreOffice Draw 経由の品質想定

LibreOffice は PDF を Writer ではなく **Draw で開く**(PDF をベクター画像として扱う)。Draw から `.docx` エクスポートは可能だが、**Word PDF Reflow より品質は低い**ことが一般的に知られている。

パターン C/D の PDF 処理は失敗率高めで、レイアウト崩れも顕著になる想定。**本評価ではあえて流して比較材料にする**(LibreOffice の限界を可視化する目的)。

#### 3.6.4 スキャン PDF の事前判別

108 件の PDF にスキャン PDF(画像のみ)が混在している可能性がある。スキャン PDF の場合:

- パターン E(MarkItDown 直接): テキスト取れず、ほぼ空出力
- パターン F(Docling 直接): **Docling は OCR 機能内蔵のためテキスト取得可能性あり**
- パターン A/B/G(COM 経由): Word は PDF 内画像を画像として扱うので、テキスト取れない
- パターン C/D(LibreOffice): Draw で開くので、テキスト取れない

つまり**スキャン PDF はパターン F のみが意味のある出力を出す可能性が高い**。これは評価結果として意味のあるシグナル(「Docling の OCR が決定打になるシナリオ」)になる。

**実装上の方針**:

- 機械評価実行時に各 PDF を `pdfplumber` 等で 1〜2 ページのテキスト抽出量を確認し、`input_pdf_type` 列を付与する
  - `text`: テキストが正常に取れる(埋め込みテキスト PDF)
  - `scan`: テキストがほぼ取れない(スキャン PDF)
  - `mixed`: 一部ページが画像化されている
- このカラムを使って「PDF タイプ別 × パターン別」の集計が可能になる

#### 3.6.5 処理時間の集計上の扱い

PDF 経由(特にパターン A/B/G)は処理時間が他形式と桁違いに大きくなる可能性がある。第 4.5 節「集計と可視化」のテーブルで **PDF を独立行として扱い、平均値が他形式に引きずられないようにする**。

第 5.2 節「機械評価の全件コスト試算」も、PDF の比率と Word PDF Reflow の所要時間が読めてから再見積もりする。

## 4. 評価軸の設計

### 4.1 評価軸の三層構造

評価は以下の 3 層に分けて実施する:

**層 1: 機械計測(自動、定量)**
処理系に組み込み、ログとして自動収集する指標

**層 2: 構造評価(半自動、定量)**
出力 Markdown をパースして、構造的な正しさを測る指標

**層 3: 人の目評価(手動、定性 + 定量)**
サンプルに対して人間が見て判定する指標。1〜5 のスコア化

### 4.2 層 1: 機械計測の指標

1 ファイル 1 パターンごとに 1 レコード記録。

| 指標 | 型 | 説明 |
|---|---|---|
| pattern_id | string | A〜G のパターン識別子 |
| input_file | string | 入力ファイル相対パス |
| input_size_bytes | int | 入力ファイルサイズ |
| input_extension | string | 拡張子 |
| input_pdf_type | string | PDF の場合のみ: `text` / `scan` / `mixed`(第 3.6.4 節)。それ以外は空 |
| input_encoding_detected | string | エンコーディング検出結果(UTF-8 / CP932 / Shift-JIS 等)。判別不可は `unknown` |
| preprocess_time_sec | float | 前処理時間(直接変換時は 0) |
| preprocess_success | bool | 前処理成功可否 |
| preprocess_error | string | 前処理エラーメッセージ |
| preprocess_timeout | bool | タイムアウトで打ち切ったか |
| intermediate_size_bytes | int | 中間ファイルのサイズ(前処理ありの場合) |
| convert_time_sec | float | 本変換時間 |
| convert_success | bool | 本変換成功可否(失敗判定基準は第 13.4 節) |
| convert_error | string | 本変換エラーメッセージ |
| convert_timeout | bool | タイムアウトで打ち切ったか |
| total_time_sec | float | 合計処理時間 |
| output_size_bytes | int | 出力 Markdown のサイズ |
| output_char_count | int | 出力文字数 |
| output_failure_reason | string | 失敗または準失敗時の分類(空出力 / 文字化け過多 / 構造崩壊 / その他、第 13.4 節) |
| tool_version | string | 使用したツールのバージョン(MarkItDown / Docling / Office / LibreOffice) |

これらから**平均処理時間、失敗率、本番規模(12 万ファイル)換算時間**が算出できる。

**カラム追加の補足**:

- `input_pdf_type`: PDF 全パターン処理(第 3.6 節)に対応。スキャン PDF とテキスト PDF を分けて評価するため必須
- `input_encoding_detected`: 旧形式(.doc / .xls / .rtf)で日本語の文字化けが疑われる場合の切り分け用。`chardet` などで判別
- `preprocess_timeout` / `convert_timeout`: COM・LibreOffice ハング時の打ち切り情報。失敗率分析の精度を上げる
- `output_failure_reason`: `convert_success` が True でも実質失敗(0 バイト出力など)を分類するための列。詳細は第 13.4 節「失敗判定基準」参照
- `tool_version`: Docling のニューラルモデル更新による結果差異を遡及できるよう記録

### 4.3 層 2: 構造評価の指標

出力 Markdown をパースして自動計測する指標。

**第 1 段階では、以下の指標のうち「ノイズ除去に依存しないもの」だけを計測**する。`normalized_char_count` や `noise_ratio` のように正規化を前提とする指標は、ノイズ除去を有効化する第 2 段階で追加する。

| 指標 | 型 | 説明 | 評価観点 | 第 1 段階 |
|---|---|---|---|---|
| heading_count | int | 見出しの総数(`#`〜`######`) | 構造保持度 | 計測 |
| heading_max_depth | int | 最大見出し階層 | 同上 | 計測 |
| heading_hierarchy_valid | bool | 階層飛び(h1→h3 等)の有無 | 構造保持度 | 計測 |
| table_count | int | Markdown テーブル数 | 表保持度 | 計測 |
| table_row_count | int | 全テーブルの行数合計 | 同上 | 計測 |
| image_ref_count | int | 画像参照 `![...]()` の数 | 画像情報の残存度 | 計測 |
| list_count | int | 箇条書きブロック数 | 構造保持度 | 計測 |
| code_block_count | int | コードブロック数 | | 計測 |
| noise_nan_count | int | `NaN` / `nan` 文字列の出現数 | ノイズ量 | **計測のみ(除去せず)** |
| noise_empty_line_ratio | float | 空行の比率 | ノイズ量 | **計測のみ(除去せず)** |
| noise_control_char_count | int | 制御文字の数 | ノイズ量 | **計測のみ(除去せず)** |
| noise_zero_width_char_count | int | ゼロ幅・BOM 等の数 | ノイズ量 | **計測のみ(除去せず)** |
| noise_replacement_char_count | int | 文字化け `�` の数 | エンコーディング適合性 | **計測のみ(除去せず)** |
| normalized_char_count | int | ノイズ除去後の文字数 | 実効情報量 | 第 2 段階で追加 |
| noise_ratio | float | (元文字数 - 正規化後) / 元文字数 | ノイズ比率 | 第 2 段階で追加 |

**第 1 段階の方針**:
- ノイズ系の指標は「**計測するが、除去はしない**」
- 各ツールが出力する素のノイズ量が数値として見えることが、ツール選定の重要な判断材料になる
- ノイズ除去ロジックは用意しておくが、第 1 段階では無効化(第 8 章参照)

### 4.4 層 3: 人の目評価の指標

各形式から代表サンプルを数件選び、同じサンプルの 7 パターン出力を並べて評価する。

**必須評価項目**(5 段階スコア):

| 指標 | 説明 |
|---|---|
| score_readability | 読みやすさ。Markdown として成立しているか |
| score_structure | 見出し・段落の構造が元文書を反映しているか |
| score_table_quality | 表が正しく Markdown テーブルになっているか |
| score_image_handling | 画像の扱いが妥当か(キャプションの残存、プレースホルダの体裁) |
| score_noise_level | ノイズ・文字化け・余分な記号がないか(5=クリーン) |
| score_information_loss | 元文書から失われた情報はないか(5=ほぼロスなし) |
| score_overall | 総合スコア |
| comment | 自由記述。特筆すべき長所短所 |

**任意追加項目**(気になったときだけ記入):

| 指標 | 説明 |
|---|---|
| score_rag_readiness | RAG に投入した際に LLM が意味を取れる品質か(5=問題なし、1=構造が崩れすぎて使えない)。第 1 段階での気づきレベルのメモ用途 |

`score_rag_readiness` の位置づけ:
- **必須ではなく、気になった項目にだけ記入**する任意欄
- WS2 の本格的 RAG 評価の先取り情報として記録しておく
- 例: 「この出力は見出しが消えているので、階層チャンキングには使えなさそう」といった気づきを残せる

評価シートの形式(Excel or CSV):

```
| pattern_id | sample_file | readability | structure | table_quality | ... | rag_readiness | comment |
|---|---|---|---|---|---|---|---|
| A          | sample_01.doc | 4 | 5 | 3 | ... | 4 | 表の罫線崩れ |
| B          | sample_01.doc | 4 | 4 | 5 | ... | 5 | 綺麗    |
| C          | sample_01.doc | 3 | 3 | 2 | ... |  | 変換で文字化け |
| ...        | ...         | ... | ... | ... | ... | ... | ... |
```

## 5. サンプリング戦略

### 5.1 機械評価と目視評価で件数を分ける

**機械評価**(自動で計測できる層 1 と層 2)は **539 件全件を対象**とする。
**目視評価**(人間が行う層 3)は **代表サンプル 20〜30 件**に絞る。

理由:

- 機械評価は人間の手が動かないため、全件にしても追加コストは時間だけ(並列化可能)
- 12 万ファイルへの外挿精度が上がる。ファイルサイズ別の処理時間分布まで取れれば、外挿に信頼区間がつけられる
- 目視評価は人間の時間が直接かかる。1 人で行う前提なら代表サンプルに集中する方が質が高い

### 5.2 機械評価(539 件全件)のコスト試算

- 539 ファイル × 6 パターン = **3,234 回の変換**
- 1 変換平均 5 秒と仮定すると **約 4.5 時間**
- 並列度 4 なら **約 1 時間強**
- 週末や夜間バッチに回せば十分許容範囲

### 5.3 目視評価の階層化サンプリング

全 539 件の中から、形式ごとの特徴を代表させる形でサンプルを抽出する。

| 形式 | 抽出件数 | 抽出基準 |
|---|---|---|
| .doc | 5 件 | サイズ最大 1 / 中央値 2 / 最小 1 / 無作為 1 |
| .docx | 5 件 | 同上 |
| .xls | 3 件 | サイズ最大 1 / 中央値 1 / 最小 1 |
| .xlsx | 3 件 | 同上 |
| .pdf | 5 件 | サイズ最大 1 / 中央値 2 / 最小 1 / 無作為 1 |
| .rtf | 3 件 | 容量大のため注意 |
| .txt | 2 件 | |
| .bik / .bpg / .bca / .bci | 4 件 | 独自形式は全件目視 |
| **合計** | **30 件** | |

**30 件 × 6 パターン = 180 回の目視**、1 サンプル 5 分として約 15 時間。分散して実施(1 日 1〜2 時間 × 1〜2 週間)。

件数は運用しながら調整可能。最初は 30 件で始めて、傾向が見えたら追加する・減らす判断ができる。

### 5.4 サンプル抽出の自動化

サンプリングロジックはコードに組み込み、**コマンド 1 発でサンプルフォルダを生成**できるようにする。これにより:

- 再現性が確保される
- サンプル件数を動的に変更できる
- 新しい形式が追加されても拡張しやすい

## 7. 人の目評価の運用設計

### 7.1 評価者

**一次評価**: あなた自身
**二次評価**(余裕があれば): チーム内の別のエンジニア、業務担当者(5 分で済む形式で)

### 7.2 評価手順

1. `reports/human_eval/eval_sheet_*.csv` を Excel で開く
2. 各行は「1 サンプル × 1 パターン」
3. サンプルファイルを開き、`output/pattern_X/` の Markdown を並べて比較
4. 5 段階スコアと自由記述を入力
5. 保存

### 7.3 ブラインド化は行わない

パターン ID を隠した状態で評価する**ブラインド化は実施しない**。

理由:
- 機械評価(層 1・層 2)で定量的な比較は別途取れているため、人の目評価は実用査定の位置づけで十分
- ブラインド化には評価用 CSV の並び替えや ID マスキングの仕組みが必要で、運用コストが増える
- 「どのパターンの結果か分かった状態で読んでも、実用上使えるかどうか」を見る方が実戦的

### 7.4 評価時間の目安

- 1 サンプル × 1 パターン: 5 分
- 30 サンプル × 7 パターン = 210 回 × 5 分 = **約 17.5 時間**
- 分散して実施(1 日 1〜2 時間 × 2 週間程度)
- 途中で傾向が見えたら早期終了判断もあり(全件やり切る必要はない)
- パターン G(COM 直接)は実装の完成度に依存するため、G だけ後回しで評価する運用もあり

## 8. ノイズ除去の設計

### 8.1 第 1 段階の方針: ノイズ除去を行わない

**第 1 段階の評価では、ノイズ除去は一切行わない**。各ツールの**素の出力**をそのまま評価対象とする。

理由:

- ツール本来の出力品質を直接比較できる(前処理バイアスが入らない)
- 「どのツールがノイズを出しやすいか」自体が評価情報になる
- ノイズ除去を適用すると「ツールは汚いが正規化で差が消えた」という現象が起き、ツール選定の判断を誤らせうる
- 生データを一度取っておけば、後で正規化を有効化したときの差分比較も可能

### 8.2 コード上の扱い: 実装は用意するが無効化

ノイズ除去ロジック自体はコードとして用意する。ただし**第 1 段階では呼び出さない**。
- 将来 ON にしたいときに設定ファイル側のフラグで切り替えられるようにする
- 除去ルールの記述やコメントは残し、設計意図が消えないようにする

`config/noise_rules.yaml`(将来用のひな形):

```yaml
# ノイズ除去ルール定義
# enabled: false で全体を無効化(第 1 段階のデフォルト)
enabled: false

# 各ルールは個別に enabled を指定可能
# 将来有効化したい場合はここをまとめて変更する
rules:
  # --- 段階 1: 自明なノイズ ---
  # NaN / nan 文字列の除去
  # 主に Excel 由来(空セルが nan として出てくる)
  - name: remove_nan_strings
    enabled: false
    pattern: "\\b[Nn]a[Nn]\\b"

  # ゼロ幅スペース、BOM、結合文字
  - name: remove_zero_width
    enabled: false

  # 制御文字(0x00〜0x1F、例外は \n \t)
  - name: remove_control_chars
    enabled: false

  # 文字化け U+FFFD
  - name: remove_replacement_char
    enabled: false

  # --- 段階 2: 整形 ---
  # 連続する空行を 1 行に圧縮
  - name: collapse_empty_lines
    enabled: false

  # 行末の余分な空白
  - name: trim_trailing_spaces
    enabled: false

  # Markdown テーブル前後の余分な改行
  - name: normalize_table_spacing
    enabled: false

  # --- 段階 3: 構造ノイズ ---
  # 「ページ 1」「ページ 2」のヘッダーフッター痕跡
  - name: remove_page_markers
    enabled: false

  # 「- -」等の意味のない罫線残骸
  - name: remove_decorative_lines
    enabled: false

  # Pandoc 風の属性アノテーション {: .table}
  - name: remove_pandoc_attrs
    enabled: false
```

コード側(`src/noise/normalizer.py` 相当):

```python
def normalize(text: str, config: NoiseConfig) -> str:
    """
    第 1 段階: config.enabled=false のため、素通り。
    将来 config.enabled=true にすると各ルールが適用される。
    """
    if not config.enabled:
        return text  # 素の出力をそのまま返す

    # 以下、将来の実装(記述は残す)
    # for rule in config.rules:
    #     if rule.enabled:
    #         text = apply_rule(text, rule)
    return text
```

### 8.3 評価指標との関係

第 4.3 節で定義したノイズ系の**構造評価指標**(`noise_nan_count`、`noise_empty_line_ratio` 等)は、**計測だけ行い、除去はしない**。

これにより:
- 各ツールの素の出力に「どの程度ノイズが含まれているか」が数値として見える
- 「ツール X はノイズが多いが情報は豊富」「ツール Y はクリーンだが取りこぼしがある」等の質的な違いが可視化される
- ツール選定の判断材料として、ノイズ量を 1 つの軸として扱える

### 8.4 出力ファイルの構成

第 1 段階では:

- `output/pattern_X/*.md`: ツールの素の出力(ノイズ除去なし)

将来、ノイズ除去を有効化した際の構成(ひな形として記述を残す):

- `output/pattern_X/raw/*.md`: ツールの素の出力
- `output/pattern_X/normalized/*.md`: ノイズ除去後の出力

この 2 段階構成に**いつでも切り替えられる**よう、ディレクトリ分割とファイル命名のロジックは設計段階で織り込んでおく。

### 8.5 第 2 段階以降への引き継ぎ

ノイズ除去を有効化するのは、以下のタイミングを想定:

- 第 1 段階の評価が完了し、採用ツールが絞れたとき
- 採用候補のツールに特徴的なノイズが明確になり、除去ルールの優先度が決まったとき
- 実運用に向けた品質調整フェーズ

そのとき `config/noise_rules.yaml` の `enabled: true` に切り替え、該当ルールも個別に有効化する。**コード変更なしで第 2 段階に移行できる**ことが、この設計の肝。

## 10. 組み合わせの実験余地

### 10.1 現在定義の 7 パターン以外の組み合わせ案

運用中に出てくる可能性:

| 拡張候補 | 想定パターン | 追加コスト |
|---|---|---|
| Pandoc converter | COM → Pandoc、LibreOffice → Pandoc | 1 converter 追加 |
| unstructured library | Direct → unstructured | 1 converter 追加 |
| OCR 前処理(スキャン PDF 用) | OCR → MarkItDown、OCR → Docling | 1 preprocessor 追加 |
| MarkItDown + Docling のアンサンブル | 両方実行し良い方を採用 | 別レイヤーの仕組みが必要 |
| python-docx / openpyxl による自前 Markdown 化 | Direct → python_docx | 実装コスト大、MarkItDown 内部と近いため優先度低 |

**設定ファイル駆動**の設計にしているため、どの追加も**コード最小限で試せる**。これは「比較評価そのものを進化させる」能力の重要性。

### 10.2 部位的な組み合わせの例

- **前処理だけ COM、後処理だけ Docling** = パターン B(既に計画内)
- **前処理だけ LibreOffice、後処理は MarkItDown と Docling 両方試す** = パターン C と D(既に計画内)
- **前処理なし、MarkItDown と Docling を両方試す** = パターン E と F(同上)
- **COM 内で前処理と本変換を一体実行** = パターン G(既に計画内)

組み合わせの自由度を最初から持っているので、**特別な拡張作業なしに 7 パターン全て既にカバー**されている。

## 11. 評価結果の使い道

### 11.1 判断フレーム

評価結果を以下 3 軸で判定:

1. **速度**: 12 万ファイルに対して現実的か(例: 並列 4 で 24 時間以内)
2. **品質**: 人の目評価で平均 3.5 以上、致命的失敗が少ない
3. **保守性**: 引き継ぎ先でトラブル対応できるか(COM は Office 依存、LibreOffice は比較的シンプル)

### 11.2 想定される採用シナリオ

**シナリオ 1: LibreOffice + Docling が優秀**
→ WSL 完結で統一。保守性最高。
→ パイプラインはシンプル(既存 graphrag-api の中で処理)

**シナリオ 2: COM + MarkItDown が抜けて優秀**
→ 精度最優先の採用。Windows 連携が必要に。
→ Windows 側に小型 FastAPI サービスを立て、WSL から HTTP 呼び出し

**シナリオ 3: 形式ごとに使い分けが最適**
→ 「.docx は MarkItDown、.doc は COM+MarkItDown、.xls は LibreOffice+Docling」等
→ 分岐ロジックがやや複雑になるが、設定ファイルで表現可能

**シナリオ 4: パターン G(COM 直接 Markdown)が飛び抜けて優秀**
→ Office ネイティブの解釈力が必須とわかったケース。Word フィールドや独自スタイルの認識が他ツールでは追いつかない
→ Windows 連携前提。ただし COM 直接実装の保守コストは大きいため、実装品質の見極めが必要
→ このシナリオの場合、採用前に独自実装の長期保守可能性を別途検討

**シナリオ 5: どれも決定的でない**
→ 最も保守性の高い LibreOffice + MarkItDown を暫定採用、本番運用で継続的に改善
→ 「決めきれなかった」ではなく「運用で育てる」という正統な判断

## 12. 説明ポイント

1. **7 パターンを系統立てて比較**する設計(前処理 3 種 × 本変換 2 種 + COM 一体型)
2. **機械的評価 + 人の目評価**の二層で偏りを排除
3. **機械評価は 539 件全件、目視評価は代表 30 件**で、統計精度と人的コストを両立

6. 評価結果は**採用判断の根拠**として提示、決定自体は関係者の合意で行う
7. **Windows 環境で完結**させ、既存 WSL/Docker パイプラインとは独立した実験として実施(採用後の統合は別タスク)
8. **並行して** WS3(ファイル関係抽出)も進めることで、評価結果を待たずにユースケース検証の道も確保

## 13. PoC 環境前提と運用

本評価は **PoC(概念実証)段階** であり、現状手元にある 539 ファイル(79.7MB)で各パターンを比較する。本番想定の 12 万ファイル規模への適用は、PoC で採用ツールが決まった**後の別タスク**として扱う。本章では PoC 段階で遵守すべき環境前提と運用ルールを整理する。

### 13.1 ライブラリ・モデルの調達(USB 移送)

air-gapped 環境のため、必要なライブラリとモデルファイルは**オンライン環境で取得し、USB 経由で評価環境に移送**する。

**初回投入手順(README に記載)**:

1. オンライン環境(別 PC)で以下を取得:
   - MarkItDown / Docling / pywin32 等の wheel ファイル一式(依存関係込み)
   - Docling のレイアウト解析・表構造抽出モデルファイル(初回ダウンロードされる ML モデル)
   - 必要に応じて `pdfplumber`(PDF タイプ判別用)
2. ウイルスチェック(社内ポリシーに従う)
3. USB で評価環境に持ち込み、所定の場所に展開
4. オフラインインストール: `pip install --no-index --find-links <wheel フォルダ> <package>`
5. Docling のモデルパス設定(環境変数または config)

**運用上の注意**:

- ライブラリやモデルのバージョンは **`tool_version` カラム**(第 4.2 節)に記録し、後で結果差異の遡及ができるようにする
- 本番化フェーズで再度バージョンを上げる場合の更新手順も同 README に追記する想定

### 13.2 日本語エンコーディング

旧形式(.doc / .xls / .rtf)は **CP932 / Shift-JIS** で書かれている可能性が高い。文字化けが発生すると「ツールの素の品質」と「環境設定不足」が切り分けられないため、以下を遵守する:

- **スモークテスト**: 評価開始前に日本語サンプル 1〜2 件で全パターンを通し、文字化けがないことを確認する
- **LibreOffice 経由(C/D)で日本語フォントが必要な場合**は、Windows 標準フォント(MS ゴシック等)で動作するか確認
- 機械評価レポートに **`input_encoding_detected` カラム**(第 4.2 節)を追加し、エンコーディング情報を残す

### 13.3 タイムアウトとマクロ・破損ファイル対策

PoC 規模(539 件)でも、以下は確実に発生するため最低限の対策を入れる:

#### 13.3.1 タイムアウト

- 1 ファイルあたりのタイムアウト値(暫定: **300 秒**、PDF パターンは別枠で **600 秒**)を設定
- タイムアウト時はプロセス強制終了し、`preprocess_timeout` / `convert_timeout` を True で記録
- 大規模化(本番化)時はこの値の妥当性を再検討

#### 13.3.2 Office 信頼センター設定(COM 経由)

評価実行前に **Word / Excel の信頼センター**で以下を設定:

- マクロを警告なしで無効化(`Application.AutomationSecurity = msoAutomationSecurityForceDisable`)
- 保護ビューを無効化(評価対象は社内ファイルなのでセキュリティリスクは管理外)
- 「Word は PDF を編集可能な Word 文書に変換します」ダイアログ抑制(第 3.6.2 節)

#### 13.3.3 パスワード保護・破損ファイル

- パスワード入力ダイアログ検出時は即 failure 扱い、`output_failure_reason = "password_protected"`
- 破損ファイル修復ダイアログも即 failure 扱い、`output_failure_reason = "corrupted"`
- COM のダイアログ自動応答ロジックを preprocessor に組み込む

### 13.4 失敗判定基準

`convert_success: bool` だけでは捉えきれないグレーゾーンを明示する。出力 Markdown が以下に該当する場合は **`output_failure_reason` で分類**する(`convert_success` は実行が完走したかどうかのみを表す):

| 判定 | 条件 | 分類値 |
|---|---|---|
| 空出力 | `output_char_count < 10` | `empty_output` |
| 文字化け過多 | `noise_replacement_char_count` が出力文字数の 50% 超 | `mojibake_majority` |
| 構造崩壊 | 見出し 0 件 + テーブル 0 件 + 段落 1 件未満 | `structure_collapsed` |
| パスワード保護 | 暗号化検出 | `password_protected` |
| ファイル破損 | 修復ダイアログ等 | `corrupted` |
| タイムアウト | 設定値超過で打ち切り | `timeout` |

集計時(第 4.5 節)は `convert_success=True` かつ `output_failure_reason` が空のものを「実質成功」として扱う。

### 13.6 本番化フェーズで再検討する項目

PoC では先送りするが、本番化(12 万ファイル展開)時に必須となる検討項目を記録しておく。

- **中間ファイルの容量管理**: 12 万件 × 中間 .docx で数十 GB 規模になる可能性、削除タイミング設計
- **Docling の再現性**: モデル更新時の評価結果差異の管理
- **目視評価ルーブリックの厳密化**: 複数評価者運用時のスコア定義明文化

- **採用ツールに合わせた既存 graphrag-api への組み込み**: Windows 連携(FastAPI / 定期バッチ)の設計
- **エンドツーエンド処理時間の最終見積もり**: 並列度・ハードウェア・キャッシュ戦略を含む
