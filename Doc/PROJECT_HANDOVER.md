## 1. プロジェクトの目的と現状

### 1.1 何をやろうとしているか

レガシー基幹システム(YPS/COBOL)の暗黙知を AI で継承する仕組みを作る。

具体的なユースケースとして上司から提示されているのは:

- **法改正対応の影響調査**(例: 消費税法改正で影響するプログラム・コード箇所を特定)
- **専門用語解説**(社内独自用語の説明)
- **エラーコード解説**(発生箇所と対処法)
- **設計書要約**

これらに自然言語で問い合わせて、AI が回答する仕組みを目指す。

### 1.2 制約

- **air-gapped 環境**(外部 NW アクセス不可)
- **ローカル LLM のみ使用可**(中国製を除く制約あり)
- **試し範囲: 539 ファイル / 79.7MB**(全体の約 0.5%)
- **本番規模: 設計書 12 万ファイル 1.2GB + ソース 1GB**
- 引き継ぎ先は AI 非専門の業務担当者(保守性が決定的に重要)

### 1.3 現在地(2026-04-25 時点)

3 つのワークストリーム(WS1, WS2, WS3)に分割し、それぞれ**設計フェーズ完了**。実装着手準備が整っている状態。WS1 は PoC 段階の運用前提も明文化済み。

```
WS1: 前処理ツール比較   ← 設計完了、実装未着手
WS2: RAG 手法比較       ← 設計完了、実装未着手
WS3: ファイル関係抽出   ← 設計完了、実装未着手
```

既存インフラ(`graphrag-api`)は構築済みで、3 つのワークストリームはこれをベースに進める。

## 2. 3 つのワークストリームの全体像

### 2.1 全体構造図

```
┌─────────────────────────────────────────────────────────┐
│               全体ゴール: 暗黙知継承 AI                  │
└─────────────────────────────────────────────────────────┘
                          ↑
        ┌─────────────────┼─────────────────┐
        │                 │                 │
   ┌────┴────┐      ┌─────┴─────┐    ┌─────┴─────┐
   │   WS1   │      │    WS2    │    │    WS3    │
   │ 前処理  │ ───→ │RAG 手法  │←──│ファイル関係│
   │ ツール  │      │ 比較評価 │    │ 抽出      │
   │ 比較    │      │           │    │           │
   └─────────┘      └───────────┘    └───────────┘
```

### 2.2 各ワークストリームの役割

#### WS1: 前処理ツール比較評価

**目的**: ドキュメント(.doc, .docx, .xls, .xlsx, .pdf, .rtf 等)を Markdown 化する複数ツールを比較し、採用判断の根拠を得る。

**比較対象**: 7 パターン
- パターン A: COM + MarkItDown
- パターン B: COM + Docling
- パターン C: LibreOffice + MarkItDown(LibreOffice 利用可否次第)
- パターン D: LibreOffice + Docling(同上)
- パターン E: Direct + MarkItDown
- パターン F: Direct + Docling
- パターン G: COM 直接 Markdown 化

**PDF の扱い**: **全 7 パターンで PDF を処理対象**とする。Word PDF Reflow(A/B/G)、LibreOffice Draw 経由(C/D)、直接変換(E/F)を比較し、Reflow が下流ツールの Markdown 品質を改善するかを評価する。スキャン PDF はパターン F(Docling)のみが意味のある出力を出す可能性あり。

**実行環境**: Windows 完結(独立した実験プロジェクト)。ライブラリ・モデルは USB 移送(air-gapped 対応)

**位置付け**: PoC 段階。タイムアウト・並列化・Docling 再現性などは PoC では暫定値で進め、本番化フェーズで再検討する(設計書 第 13.6 節)

**成果物**: 各パターンの変換結果 + 評価レポート(機械評価 539 件全件 + 目視評価 30 件)

**詳細**: `conversion_eval_design.md`

#### WS2: RAG 手法比較評価

**目的**: 3 つの RAG 手法を比較し、本プロジェクトで採用する RAG アーキテクチャを決定する。

**比較対象**: 3 手法
- 手法 A: 既存 GraphRAG(チャンク + ベクトル + Neo4j 知識グラフ)
- 手法 B: Dify 標準ベクトル RAG(シンプルなベクトル検索)
- 手法 C: 有識者組織流「全文送信」方式(関係グラフ + 関連ファイル群を丸ごと LLM へ)

**評価構造**: 2 段階
- 段階 1: 同一 Markdown × 3 手法(RAG 手法そのものの素性比較)
- 段階 2: 複数前処理パターン × 3 手法(相性評価)

**質問セット**: 4 カテゴリ × 5 問 = 20 問(事実検索 / 影響範囲 / 概念説明 / 設計書要約)

**詳細**: `rag_evaluation_design.md`

#### WS3: ファイル関係抽出

**目的**: COBOL ソース間の呼び出し関係(CALL/COPY)を Neo4j に機械的に抽出する。WS2 手法 C の基盤としても使用する。

**対象**:
- COBOL ソース(.cbl / .cob / .cpy)からの CALL/COPY 抽出
- COBOL ⇔ 設計書の紐付け(将来)

**実装方針**: 既存 `graphrag-api` に追加機能として組み込む(別ラベル `SourceFile`/`CopyBook`/`DesignDoc` で既存と分離)

**ユースケース**: 消費税影響調査(例: 「消費税の改正で影響するプログラムは?」→ 関連ソースを呼び出し関係から辿って提示)

**詳細**: `cobol_relations_design_memo.md`

### 2.3 ワークストリーム間の依存関係

```
WS1(前処理)──→ WS2(RAG 評価)
                    │
                    └──→ 段階 2 で WS1 の上位パターンを使う

WS3(関係抽出)──→ WS2(RAG 評価)の手法 C
                    │
                    └──→ WS3 の SourceFile グラフが手法 C の基盤
```

**並行実施できる部分**:
- WS1 と WS3 は完全独立(同時進行可能)
- WS2 のフェーズ 1〜3(質問セット作成、手法 C プロトタイプ実装)は WS1/WS3 の途中でも着手可能

## 3. 既存資産

### 3.1 既存インフラ

#### `graphrag-api`(WSL の Docker で稼働中)

FastAPI ベースの自社 GraphRAG 実装。**WS3 の追加機能の母体**になる。

**主要エンドポイント**:
- `/ingest`: 単一ドキュメントの取り込み(チャンキング → ES + Neo4j 投入)
- `/ingest-temp`: 一時ファイル(TTL 24h)の取り込み
- `/ingest-dir`: 入力フォルダの一括取り込み(バッチ、`/input` 配下)
- `/ingest-growi`: Growi ページの取り込み
- `/search`: ベクトル検索 + 知識グラフ展開
- `/documents`: ドキュメント一覧、ES と Neo4j の整合性確認
- `/documents/{id}`: 削除、再取り込み
- `/ui`: 管理 UI(ファイルアップロード、再同期、削除)

**コード構成**:
- `main.py`: API エンドポイント、ingest/search のメインロジック
- `ingest.py`: ドキュメント形式別の payload 構築(.pdf/.md/.txt 対応)
- `providers.py`: 埋め込み・LLM のプロバイダ抽象(Bedrock / Gemini / Ollama 切替可)
- `docker-compose.yml`: 全サービス定義
- `reset_index.sh`: ES と Neo4j を一括リセット

#### 既存 Neo4j スキーマ

```
(:Document {id, title, url, scope, expires_at, ...})
  -[:HAS_CHUNK]->
(:Chunk {id, text, chunk_index, ...})
  -[:MENTIONS]->
(:Entity {canonical_name, name, type})
  -[:RELATED_TO {relation_type, source_document_id}]->
(:Entity)
```

**重要な設計パターン**: `RELATED_TO.source_document_id` で関係の由来を追跡し、再取り込み時に自分が作った関係だけクリーンに消せる。**WS3 でも同パターンを踏襲**(`CALLS.source_file` 等)。

#### 既存 Elasticsearch インデックス

- インデックス名: `graphrag_chunks`
- 主要フィールド: `chunk_id`, `document_id`, `text`, `embedding`, `category`, `scope`, `expires_at` 等
- `category` は `path_hierarchy` tokenizer で階層検索可能

### 3.2 動いている管理 UI

`graphrag-api` の `/ui` で:
- ドキュメント一覧表示(scope = official / temporary / all で切替)
- 一時ファイルアップロード(TTL 付き)
- `/ingest-dir` の起動と進捗表示
- ドキュメントの削除・再取り込み
- ES と Neo4j の整合性チェック(mismatch 表示)

### 3.3 LLM プロバイダ構成

`providers.py` で 3 系統を切替可能:
- Bedrock: Titan 埋め込み、Claude Haiku
- Gemini: text-embedding-004、Gemini 2.0 Flash
- Ollama: nomic-embed-text、各種ローカル LLM

本プロジェクトでは **Ollama 中心**(air-gapped 制約のため)。Gemma 4(256K)が候補に挙がっている。

## 6. 重要な制約と判断履歴

### 6.1 環境制約

- **air-gapped**: 外部 NW アクセス禁止
- **ローカル LLM 限定**: 中国製を除く(2026-04 時点で Gemma 4 が有力候補、256K コンテキスト)
- **ドキュメント機密性**: 社外への持ち出し不可
- **実行環境分離**: WS1 は Windows 完結、WS2/WS3 は WSL 上の既存インフラを使用

### 6.2 主要な判断履歴

#### WS1 関連

- **比較対象を 7 パターンに拡張**: 当初 6 パターンだったが、「COM 直接 Markdown 化」をパターン G として追加(Office ネイティブの解釈力を上限品質として測る目的)
- **第 1 段階ではノイズ除去をしない**: ツール本来の素の出力で比較するため。除去ロジックは将来用に用意するが無効化
- **機械評価は全件、目視評価はサンプル**: 機械評価 539 件 + 目視評価 30 件で精度とコストを両立
- **Windows 環境で完結**: 既存 WSL/Docker パイプラインとは独立した実験プロジェクトとして扱う
- **COBOL ソースは WS1 対象外**: 設計書系と分離、WS3 で別設計
- **PDF は全 7 パターンで処理する(2026-04-25 追加)**: 当初は直接変換(E/F)のみの方針だったが、Word PDF Reflow / LibreOffice Draw 経由が下流の Markdown 品質を改善するかを比較するため、全パターンで処理対象に拡張(設計書 第 3.6 節)
- **PoC として位置付ける(2026-04-25 追加)**: 12 万ファイル規模はあくまで本番想定。PoC 段階では USB 移送によるライブラリ調達、暫定タイムアウト値、目視評価は 1 人運用で進め、並列化や Docling モデル再現性の本格設計は本番化フェーズで再検討(設計書 第 13 章)
- **失敗判定基準の明文化(2026-04-25 追加)**: `convert_success` のグレーゾーン(空出力・文字化け過多・構造崩壊・パスワード保護・破損・タイムアウト)を `output_failure_reason` で分類し、レポート集計時に「実質成功」を切り出せるようにする(設計書 第 13.4 節)
- **Markdown の自然言語化(verbalization)は WS1 対象外(2026-04-25 追加)**: 表やリストを LLM で散文に変換する後処理は、ツール本来の品質と LLM 効果の切り分けが不能になり、評価コストも 2 倍になるため WS1 では扱わない。WS2 の RAG 評価で「投入形式」として比較する(WS1 設計書 第 3.4 節 / WS2 設計書 第 3.5 節)

#### WS2 関連

- **2 段階評価構造**: 段階 1 で同一 Markdown × 3 手法、段階 2 で前処理パターンとの相性を評価
- **質問は 4 カテゴリ × 5 問 = 20 問**: 事実検索、影響範囲、概念説明、設計書要約
- **チャンキング戦略は WS2 のスコープ外**: 変換結果が出てから別途検討する
- **正解データは簡易方式**: 「キー情報リスト」を事前作成、業務担当者を巻き込まない前提
- **Markdown 自然言語化(verbalization)を追加評価軸として組み込み(2026-04-25 追加)**: 段階 1 完了後の追加検証(案 B)として、有望手法 1〜2 つに絞り「Markdown 直接投入 vs 自然言語化後投入」を比較。フェーズ 4.5 として 2〜3 日見込み(設計書 第 3.5 節)

#### WS3 関連

- **既存 Neo4j に相乗り**: 別ラベル(`SourceFile`、`CopyBook`、`DesignDoc`)で領域分離、新インフラ追加なし
- **正規表現で機械抽出**: LLM は使わない、決定論的で高速、保守可能
- **`source_file` プロパティで由来追跡**: 既存 `RELATED_TO.source_document_id` パターンを踏襲
- **設計書紐付けは第 2 週以降**: 第 1 週は CALL/COPY のみ(段階的拡張)

### 6.3 アーキテクチャ上の重要な原則

#### 原則 1: 既存と新規を物理的に分離

新機能(WS3 のファイル関係グラフ)は、既存の Document/Chunk/Entity スキーマを一切汚さない。Neo4j のラベルを完全に分け、関係名も被らせない。

#### 原則 2: 引き継ぎを前提とした保守性優先

引き継ぎ先は AI 非専門の業務担当者であるため:
- 設定ファイル駆動(コード変更なしで挙動を変えられる)
- 機械抽出を優先(LLM 確率挙動への依存を最小化)
- ドキュメント化を徹底(なぜこう決めたかも残す)

#### 原則 3: 段階的な実装と評価

最初から完璧を目指さず、最小限の動作確認 → 段階的に拡張。WS3 は 5 日プラン、WS1 は 7 フェーズで段階的に進める設計。

## 7. 検討から外したもの(なぜ捨てたかの記録)

引き継ぎ時に「なぜこれを採用しなかったのか」を残しておくことは、再検討の手戻りを防ぐ。

### 7.1 WS1 で外した選択肢

#### LibreOffice 直接 Markdown 化

LibreOffice には Markdown エクスポート機能がなく、自前実装すると COM 版とほぼ同じ実装コスト。Office ネイティブ機能(フィールド評価等)も使えないため、追加価値がないと判断。

#### python-docx / openpyxl による自前 Markdown 化

MarkItDown の内部実装と本質的に同じことになる。独自実装の意義が薄いため対象外。

#### 独自形式(.bik / .bpg / .bca / .bci)のネイティブ対応

YPS/COBOL の富士通系独自バイナリ形式。ライブラリは存在しない。今回は「**試験的にパイプラインに流して挙動を観察**」までにとどめ、深い対応は将来課題。

### 7.2 WS2 で外した選択肢

#### チャンキング戦略の比較

変換結果(WS1 の出力 Markdown)を見てから決めないと意味がない。WS2 のスコープ外として、採用後の調整課題に位置づける。

#### Dify ワークフローの質問分類実装

複数 RAG 手法を質問種別で振り分ける実装は、採用判断後の運用設計の話。本評価では各手法を独立して評価する。

### 7.3 全体方針として外した選択肢

#### 業務担当者のヒアリングを前提とした設計

業務担当者が忙しすぎて巻き込めないという制約のため、ヒアリングなしで進む方針。代わりに「業務担当者なしで動くものを作って関心を引く → ヒアリングの機会を作る」という戦略を採用。

#### 一気に全領域を統合した RAG 構築

スコープが大きすぎて 1 人で背負える範囲を超えるため、段階的に分けて評価する方針に転換。「全部並行でやる」のではなく「ワークストリームに分割し、必要に応じて並行する」形に整理。

## 8. ドキュメントの保守

### 8.1 各ドキュメントの更新タイミング

- **`conversion_eval_design.md`**: WS1 の進行に応じて、未解決事項の解消や評価結果を追記

- **本ドキュメント**: 上記 3 つに大きな変更があったとき、または進行ステータスが変わったとき

### 8.2 引き継ぎ時に確認すべきこと

新しい人がプロジェクトを引き継ぐとき、以下を順に確認するとスムーズ:

1. **本ドキュメントを通読**(全体像把握)
2. **3 つの設計ドキュメントを目次レベルで把握**
3. **既存コード(`graphrag-api`)を起動して動かす**(`/ui` でドキュメント投入を試す)
4. **進行ステータス(本ドキュメント第 5 章)を確認**して、自分が引き継ぐタスクを特定
5. **該当ワークストリームの設計ドキュメントを精読**
