# RAG 手法比較評価 - 設計ドキュメント

**作成日**: 2026-04-17
**最終更新**: 2026-04-25
**対象**: YPS/COBOL 知識継承プロジェクト WS2(RAG 手法比較評価)
**目的**: 3 つの RAG 手法を比較評価し、本プロジェクトで採用する RAG アーキテクチャの判断材料を得る

---

## 1. 背景と目的

### 1.1 なぜ RAG 手法の比較が必要か

レガシー基幹システムの暗黙知継承を目的とした本プロジェクトでは、ユーザーからの自然言語質問に対して、ドキュメント群を検索し適切な回答を生成する RAG(Retrieval-Augmented Generation)システムを構築する。

RAG にはいくつかのアーキテクチャがあり、それぞれ得意領域と制約が異なる。どれを採用するかは精度・速度・保守性に大きく影響するため、**実データで比較して判断する**ことが必要。

現在、以下の 3 手法が候補に挙がっている:

- **既存 GraphRAG**: 自社で既に構築済みの仕組み
- **Dify 標準ベクトル RAG**: Dify のナレッジベース機能を使う最もシンプルな構成
- **有識者組織流「全文送信」方式**: 社外の有識者が運用している、コンテキスト拡張時代の新しい手法

### 1.2 評価対象の 3 手法

#### 手法 A: 既存 GraphRAG

- 既に構築済みの `graphrag-api`(FastAPI + Elasticsearch + Neo4j)
- ドキュメントをチャンク化 → ベクトル化して Elasticsearch に格納
- 並行して LLM がエンティティと関係を抽出 → Neo4j に知識グラフとして保存
- 検索時は、まずベクトル類似検索で候補チャンクを取得、次に知識グラフで関連チャンクを追加取得、両方を LLM に渡して回答生成
- 本プロジェクトの現時点で最も投資が進んでいる手法

#### 手法 B: Dify 標準ベクトル RAG

- Dify のナレッジベース機能を利用
- チャンキング、埋め込み、検索、プロンプト統合まで Dify が内蔵機能として提供
- グラフなし、純粋なベクトル類似検索のみ
- 最もシンプルで保守しやすい構成

#### 手法 C: 有識者組織流「全文送信」方式

- コンテキスト長の拡大(2026 年現在、Gemma 4 で 256K 等)を前提にした新しい方式
- ファイル間の関係(参照、呼び出し、同一サブシステム等)だけをグラフ DB に持つ
- 検索時:
  1. 質問から関連するキーワードや対象を特定
  2. グラフ DB で関連ファイル群を拾う
  3. それらのファイルの **全文 Markdown を丸ごと LLM のコンテキストに入れる**
  4. LLM に回答させる
- チャンキングを極力しない(あるいは粗いチャンキングだけ)
- 本プロジェクトの制約(ローカル LLM、コンテキスト上限 128〜256K)に合わせた実装が必要

### 1.3 スコープの境界

**本評価に含むもの**:
- 3 手法の回答品質比較
- 3 手法の応答速度比較
- 3 手法の運用コスト(構築、保守、チューニング難易度)の定性評価
- WS1 で評価した複数の前処理パターンとの組み合わせ評価(第 3 章の段階 2)

**本評価に含まないもの**:
- 前処理ツールそのものの品質評価(WS1 の範疇)
- ファイル間関係抽出の実装設計(WS3 の範疇。ただし手法 C の基盤として WS3 の成果物を使用する)
- Dify ワークフロー側の UI 設計や質問分類ロジックの実装

### 1.4 WS1・WS3 との関係

本 WS2 は、他の 2 つのワークストリームと以下のように接続する:

**WS1(前処理ツール比較評価)との接続**:
- WS1 で出力される Markdown が、WS2 の入力になる
- WS2 は「RAG 手法そのもの」の評価が主目的だが、「WS1 採用候補の前処理パターンとの相性」も段階 2 で評価する
- WS1 の結果が出揃う前でも、WS2 は段階 1(同一 Markdown での比較)から先行して実施可能

**WS3(ファイル関係抽出)との接続**:
- 手法 C(全文送信方式)では、**ファイル関係グラフが検索の起点**になる
- つまり WS3 で構築する `SourceFile → CALLS → SourceFile` や `SourceFile → DESCRIBED_BY → DesignDoc` の関係グラフを、手法 C の実装で参照する
- WS3 の成果物がないと手法 C は実装できないため、**WS3 と WS2 の手法 C 実装は時間的に WS3 が先**

## 2. 3 手法の仕様整理

### 2.1 手法 A: 既存 GraphRAG の仕組み

#### 2.1.1 データ投入フロー

1. ドキュメントを `/ingest` エンドポイントで受け取る
2. `RecursiveCharacterTextSplitter` で `CHUNK_SIZE=800`、`CHUNK_OVERLAP=120` のチャンクに分割
3. 各チャンクを埋め込みモデル(Bedrock Titan / Gemini / Ollama のいずれか)でベクトル化
4. Elasticsearch にチャンク本文・ベクトル・メタデータを格納
5. Neo4j に `Document` ノードと `Chunk` ノードを作成、`HAS_CHUNK` 関係で接続
6. 各チャンクから LLM でエンティティ(canonical_name + type)を抽出、`Entity` ノードとして登録、`MENTIONS` 関係で接続
7. 抽出されたエンティティ間の関係を LLM で抽出、`RELATED_TO` 関係で接続

#### 2.1.2 検索フロー

1. 質問文を埋め込みベクトル化
2. Elasticsearch で kNN 検索、上位チャンクを取得(seed chunks)
3. seed chunks から `MENTIONS` 関係で接続する `Entity` を取得
4. その `Entity` を `MENTIONS` する別の `Chunk` を取得(graph hits)
5. seed chunks と graph hits をマージして LLM に渡す
6. LLM が回答生成

#### 2.1.3 現状の制約

- チャンクサイズ 800 はコンテキスト制約時代の値で、現在は粗くしてもよい可能性がある
- エンティティ抽出・関係抽出は LLM 任せで、結果が確率的(再現性が低い)
- 知識グラフの品質はドキュメントの性質に強く依存する

### 2.2 手法 B: Dify 標準ベクトル RAG の仕組み

#### 2.2.1 データ投入フロー

- Dify の管理画面で「ナレッジベース」を作成
- Markdown ファイルをアップロード
- Dify が自動でチャンキング、埋め込み、検索インデックス構築
- チャンキング方法(固定サイズ / セマンティック / 親子)は Dify 側で選択可能

#### 2.2.2 検索フロー

- Dify ワークフロー内の「ナレッジ検索」ノードで、質問に類似したチャンクを取得
- 取得したチャンクを LLM ノードのコンテキストに渡す
- LLM ノードが回答生成

#### 2.2.3 手法の特徴

- 構築・運用がシンプル。GUI で完結
- 知識グラフや複雑な検索ロジックはない
- チャンキング設定の調整幅は Dify が提供する範囲に限られる

### 2.3 手法 C: 有識者組織流「全文送信」方式の仕組み

#### 2.3.1 基本アイデア

コンテキストウィンドウが大きい前提(100 万トークン級)では、「チャンクで刻んで関連箇所だけ送る」より「関連ファイル群を丸ごと送る」方が精度が高い、という考え方。

ただしローカル LLM では 128K〜256K が上限なので、完全な「全ファイル丸ごと」は難しい。**折衷案**として、「関連ファイル群を特定し、**コンテキストに収まる範囲で全文を送る**」方式を取る。

#### 2.3.2 データ投入フロー

1. ドキュメントを Markdown 化(WS1 の成果物を使用)
2. ファイルごとに Markdown を**そのまま保存**(細かいチャンキングはしない)
3. ファイル間の関係を Neo4j に登録(WS3 で実装)
   - COBOL ソース間の CALL 関係
   - COBOL ソース → コピー句 の COPY 関係
   - COBOL ソース → 設計書 の DESCRIBED_BY 関係
4. 同時にキーワード全文検索用のインデックス(Elasticsearch または grep)を整備

#### 2.3.3 検索フロー

1. 質問から関連キーワードを LLM で展開(例: 「消費税」→ ["消費税", "ZEI", "TAX", "SHOUHIZEI", ...])
2. キーワードでファイル全文検索、直接ヒットするファイルを得る
3. Neo4j の関係グラフで、直接ヒットに関連するファイル(呼び出し元、対応設計書等)を最大 N ホップ追跡
4. 直接ヒットと関連ファイルの **Markdown 全文を結合**
5. コンテキストサイズ上限内に収まるよう、優先度順に切り詰め
6. LLM に渡して回答生成

#### 2.3.4 ローカル LLM 制約への対応

- Gemma 4(256K)、Llama 3.3(128K)等を想定
- 1 ファイルあたり平均 10K トークンとして、**約 10〜20 ファイル分の全文**が入る計算
- ファイル数が多いときは「直接ヒットしたファイル優先 → 関連ファイルは重要度順」で切り詰め
- 本プロジェクトの WS3 成果物(ファイル関係グラフ)がこの方式の基盤となる

#### 2.3.5 手法の特徴

- ファイルを細切れにしない分、文脈が保たれる
- 実装は「関係グラフの構築」と「コンテキスト組み立てロジック」が主で、複雑な検索チューニングは不要
- コンテキスト上限がネックになりやすい
- チャンキング戦略に悩まなくて済む

## 3. 評価の 2 段階構造

### 3.1 なぜ 2 段階にするか

前処理方式と RAG 手法の両方を同時に比較すると、変数が掛け算で増えて「何が原因で精度が落ちたか」の切り分けが不能になる。

そこで評価を 2 段階に分け、**段階 1 で RAG 手法そのものの素性を見る**、**段階 2 で前処理方式との相性を見る**、という順序で進める。

### 3.2 段階 1: 同一 Markdown × 3 手法

#### 目的

RAG 手法そのものの違い(検索アルゴリズム、知識グラフの活用、コンテキスト組み立て方)が回答品質にどう効くかを、前処理の影響を排除した条件で比較する。

#### 入力

WS1 で評価中の前処理パターンのうち、**代表として 1 つ**を選び、その Markdown を使う。選定基準:

- 機械評価で上位にあり、ノイズが少ないもの
- WS1 の目視評価が高い
- 典型的な設計書・Excel・PDF を含むサンプル

最有力候補は「パターン E: Direct + MarkItDown」または「パターン F: Direct + Docling」。新形式ファイルのみを対象にすれば、前処理のノイズ要因を最小化できる。

#### 実施手順

1. 代表 Markdown を 3 手法それぞれに投入
2. 共通質問セット(第 4 章で定義)を 3 手法それぞれに投げる
3. 各手法の回答、応答時間、引用ソースを記録
4. 回答品質を評価(第 4 章)

#### 得られる情報

- どの RAG 手法が、同じデータに対して最も高品質な回答を出すか
- 応答速度の違い
- 引用ソースの透明性(どこから答えを引いているか明示できるか)

### 3.3 段階 2: 複数変換結果 × 3 手法

#### 目的

前処理方式と RAG 手法の**組み合わせによる相性**を評価する。「この前処理はこの RAG に合う」「この組み合わせだと情報が落ちる」といった現象を検出する。

#### 入力

WS1 の段階 1 評価で上位に残った前処理パターン(2〜3 個)を候補として選び、それぞれの Markdown を使う。

例: WS1 で以下 3 パターンが上位だった場合
- パターン B: COM + Docling
- パターン D: LibreOffice + Docling
- パターン E: Direct + MarkItDown

これら 3 パターン × 3 手法 = 9 組み合わせのマトリクスで評価する。

#### 実施手順

1. 各前処理パターンの Markdown を、3 手法すべてに投入
2. 段階 1 と同じ質問セットを 9 組み合わせすべてに投げる
3. 結果をマトリクスにまとめる

#### 得られる情報

- 「前処理 X × RAG Y」の最適な組み合わせ
- 前処理品質の違いが RAG 精度にどう波及するか
- ある前処理で情報が落ちても、特定の RAG ならカバーできるか

### 3.4 段階の時間的依存関係

段階 1 は WS1 の結果が完全に出揃う前でも開始できる。代表 Markdown を 1 つ選べばよいため、WS1 の途中経過で十分。

段階 2 は WS1 の段階 1 評価(機械 + 目視)が完了し、上位候補が絞り込まれた後に実施する。

### 3.5 投入形式の追加軸: Markdown 直接投入 vs 自然言語化後投入

#### 3.5.1 背景

「Markdown を LLM で自然言語の散文に変換してから RAG に投入すると精度が上がる」というアプローチが知られている(verbalization、narrative transformation、document expansion 等の名称)。具体的には、表やリストといった構造化情報を「この表は売上推移を示しており、2023 年は…」のような散文に展開することで、embedding が意味を捉えやすくなる、という仮説。

WS1 の前処理品質評価とは独立した軸であり、**RAG 投入時の augmentation** として WS2 で扱うのが妥当(理由は WS1 設計書 第 3.4 節)。

#### 3.5.2 評価方法の候補

**案 A: 段階 1 を拡張**(同一 Markdown × 3 手法 × 投入形式 2 種類 = 6 評価)

- 段階 1 の代表 Markdown を「直接投入」と「自然言語化後投入」の 2 形式で用意
- 各手法に両方を投入し、回答品質スコアの差を見る
- 評価コストは段階 1 が 2 倍(60 → 120 質問評価)

**案 B: 段階 1 完了後の追加検証**

- 段階 1 で最も有望だった 1〜2 手法に対してのみ、自然言語化の効果を測る
- 評価コストを抑えつつ、効果のある手法に絞って深掘りできる
- 「直接投入で既に精度が高い手法」と「精度が低かった手法」の両方で試すと、自然言語化が**底上げ用**か**上限引き上げ用**かが見える

**現状の方針**: **案 B を推奨**。段階 1 で各手法の素性を見たうえで、自然言語化の効果検証を追加で行う。理由:

- 段階 1 のスコープを膨らませず、まず手法そのものの比較を優先
- 自然言語化の効果は質問種別(事実検索 / 影響範囲 / 概念説明 / 設計書要約)との相性が大きい想定で、段階 1 の結果を見てから設計した方が無駄が少ない
- 案 A の 6 評価構造は綺麗だが、自然言語化のプロンプト・モデル選定の試行錯誤が入ると評価コストがさらに増える

#### 3.5.3 自然言語化の実装方針(暫定)

- **ローカル LLM(Gemma 4 等)を使用**: air-gapped 制約下で実行可能なモデルに限定
- **プロンプトは 1〜2 種類で固定**: プロンプトチューニングまでは PoC 段階ではやらない
- **対象は表・リスト・コードブロック**: 散文部分はそのまま残す(冗長化を避ける)
- **元の Markdown も保持**: 「自然言語化後だけ」ではなく「Markdown + 自然言語化」の併存形式も比較候補

詳細な実装方針は段階 1 完了後に決める。第 9 章「未解決事項」に記載。

#### 3.5.4 質問種別ごとの効果想定

| 質問カテゴリ | 自然言語化の効果想定 | 理由 |
|---|---|---|
| 事実検索系 | **効果大** | 表の値を文章化すると embedding が拾いやすい |
| 影響範囲調査系 | 中(または逆効果の可能性) | 文章化で構造情報(参照関係)が失われるリスク |
| 概念説明系 | 中〜小 | 元 Markdown の見出し構造でも十分機能する |
| 設計書要約系 | 小 | 元の階層構造の方が要約に向く |

これは仮説であり、段階 1 の結果と合わせて検証する。

## 4. 評価軸の設計

### 4.1 質問セット

#### 4.1.1 質問の粒度区分

RAG の特性が違う領域で評価するため、質問を 4 カテゴリに分類する:

**カテゴリ 1: 事実検索系**
- 「エラーコード E001 の意味は?」
- 「プログラム TAX001 の目的は?」
- 単一ファイル内の情報を拾えば答えられる
- ベクトル検索系が強いと予想される

**カテゴリ 2: 影響範囲調査系**
- 「消費税改正の影響範囲は?」
- 「サブシステム A を停止したら、影響する処理は?」
- 複数ファイルを横断して関係を辿る必要がある
- グラフや関係情報を持つ手法が強いと予想される

**カテゴリ 3: 概念説明系**
- 「本システムにおける『締め処理』とは何?」
- 複数ドキュメントから概念を統合する必要がある
- 知識グラフや全文送信が強いと予想される

**カテゴリ 4: 設計書要約系**
- 「プログラム SUB002 の仕様を要約して」
- 特定の 1 ファイルを深く読む必要がある
- 全文送信方式が強いと予想される

#### 4.1.2 質問数の目安

- 各カテゴリ 5 問 × 4 カテゴリ = **20 問**
- 段階 1 では 20 問 × 3 手法 = 60 回の評価
- 段階 2 では 20 問 × 9 組み合わせ = 180 回の評価

180 回は多いが、1 回の実測自体は自動化できる(質問投げて回答取得するだけ)。時間がかかるのは回答の品質評価(第 4.3 節)。

### 4.2 正解データの作り方

#### 4.2.1 基本方針

業務担当者から正解を聞けないという制約下で、以下を正解の基準とする:

**簡易正解方式**: 質問を作る段階で、「回答にこの情報が含まれていれば正解」という**キーとなる情報のリスト**を事前に書き出す。

例:
- 質問: 「プログラム TAX001 の目的は?」
- 期待される情報(キー):
  - 消費税計算処理であること
  - 入力は取引金額
  - 出力は税額
  - 適用税率は外部パラメータから取得

回答がこれらのキーをすべて含めば 5 点、7 割含めば 4 点、半分程度で 3 点、といった採点基準とする。

#### 4.2.2 誰が正解を作るか

一次正解はあなた自身が作成する。業務担当外のため、ソースコードや設計書を読みながら作る必要があり、1 問あたり 10〜15 分は見込む。

20 問作成で **約 5 時間**。質問セット作成はこのコストを含めて計画する。

#### 4.2.3 正解データのフォーマット

Excel または CSV で 1 行 1 問。

```
| question_id | category | question | key_info_1 | key_info_2 | key_info_3 | ... | notes |
|---|---|---|---|---|---|---|---|
| Q001 | 事実検索 | TAX001の目的は? | 消費税計算 | 入力:取引金額 | 出力:税額 | ... | 5点満点 |
| ... |
```

### 4.3 評価指標

#### 4.3.1 回答品質(人の目評価、5 段階スコア)

| 指標 | 説明 | 採点基準の例 |
|---|---|---|
| score_correctness | 正確性。キー情報を網羅しているか | 全キー網羅=5、7割=4、半分=3、一部=2、全く外れ=1 |
| score_completeness | 網羅性。関連する補足情報も拾えているか | 期待以上=5、十分=4、最低限=3、不足=2、欠落=1 |
| score_conciseness | 簡潔性。冗長な説明で埋もれていないか | 必要十分=5、やや冗長=4、冗長=3、無関係多い=2、ノイズだらけ=1 |
| score_source_attribution | 引用の透明性。どこから答えたか分かるか | 正確に示せる=5、不正確=3、示せない=1 |
| score_overall | 総合スコア | |
| comment | 自由記述 | |

#### 4.3.2 応答時間(機械計測)

| 指標 | 型 | 説明 |
|---|---|---|
| retrieval_time_ms | int | 検索部分の処理時間 |
| llm_time_ms | int | LLM 推論時間 |
| total_time_ms | int | 合計 |
| retrieved_chunks | int | LLM に渡されたチャンク数 |
| total_tokens | int | LLM に渡された総トークン数 |

#### 4.3.3 運用コスト(定性評価)

段階 1 終了時点でまとめる運用観点の主観評価:

- 構築時間(初回セットアップにかかった時間)
- チューニング自由度(結果が悪いとき、調整できる余地)
- デバッグ容易性(なぜその回答になったか追跡できるか)
- 引き継ぎ可能性(AI 非専門者が保守できるか)

### 4.4 評価シートのフォーマット

```
| question_id | pattern_id | method | score_correctness | score_completeness | score_conciseness | score_source | score_overall | total_time_ms | comment |
|---|---|---|---|---|---|---|---|---|---|
| Q001 | pattern_e | A_graphrag | 4 | 5 | 3 | 4 | 4 | 1200 | 知識グラフで補足情報が豊富 |
| Q001 | pattern_e | B_dify    | 3 | 3 | 4 | 2 | 3 | 800  | 単純検索、補足なし |
| Q001 | pattern_e | C_fulltext| 5 | 5 | 2 | 5 | 4 | 3500 | 全文送信で情報豊富だが長い |
| ... |
```

段階 1 では `pattern_id` は 1 種類に固定。段階 2 では複数パターンが並ぶ。

## 5. 段階 1 の実施設計

### 5.1 使う Markdown の選定基準

段階 1 で使う「代表 Markdown」の選定は、以下の条件を満たすものを優先する:

1. WS1 の機械評価で**失敗率ゼロ**(変換が安定している)
2. WS1 の目視スコアで**平均 4 以上**(品質が十分)
3. **新形式ファイル(.docx/.xlsx/.pdf)のみ**で揃えられる(前処理ノイズを最小化)
4. WS1 の処理時間が**中央値程度**(極端に速い/遅いツールを避ける)

最有力候補: **パターン E(Direct + MarkItDown)**。新形式専用で、前処理なし、高速、軽量で挙動が読みやすい。

### 5.2 3 手法への投入手順

#### 手法 A(既存 GraphRAG)への投入

既存の管理 UI を使う:

1. Dify で作成した管理 UI(`/ui` エンドポイント)にアクセス
2. 選定した Markdown ファイル群を対象フォルダにアップロード
3. `/ingest-dir` エンドポイントを実行
4. ジョブ完了を `/ingest-job/{job_id}` で確認

#### 手法 B(Dify 標準ベクトル)への投入

Dify 管理画面で操作:

1. 新規ナレッジベースを作成(例: `eval_stage1_pattern_e`)
2. Markdown ファイル群をアップロード
3. チャンキング設定を標準(デフォルト)にする
4. 埋め込みモデルを選択(GraphRAG と同じにすると比較しやすい)
5. インデックス構築完了まで待つ

#### 手法 C(全文送信方式)への投入

WS3 の成果物を活用する:

1. WS3 で構築済みの Neo4j ファイル関係グラフを使用
2. Markdown ファイル群を指定フォルダに配置(FastAPI 側の `/input` と同じで OK)
3. ファイル一覧と関係グラフが対応していることを確認
4. **全文送信用の検索エンドポイント**を新規実装(第 5.4 節)

### 5.3 結果の記録フォーマット

各質問 1 問 × 3 手法の結果を、以下の形式で保存:

**raw ログ(CSV)**:
```
timestamp, question_id, method, retrieved_file_count, total_tokens_sent, llm_response, retrieval_time_ms, llm_time_ms, total_time_ms, error
```

**回答テキスト(Markdown)**:
- ディレクトリ: `reports/stage1/responses/<question_id>/`
- ファイル: `A_graphrag.md`, `B_dify.md`, `C_fulltext.md`
- 中身: 質問文、回答文、引用ソース一覧

### 5.4 手法 C の新規実装(最小プロトタイプ)

WS3 の `/impact-analysis` エンドポイントをベースに、段階 1 評価用の簡易版として `/fulltext-rag` を新設:

**入力**:
```json
{
  "question": "ユーザーの質問文",
  "max_files": 20,
  "max_tokens": 200000
}
```

**処理**:
1. 質問文から LLM でキーワード展開
2. Elasticsearch(既存 GraphRAG のインデックス流用)でキーワード全文検索、ヒットファイル一覧を取得
3. Neo4j で関連ファイルを 1〜2 ホップ追跡
4. ファイルを優先度順に並べる(直接ヒット優先)
5. Markdown ファイルの全文を結合、`max_tokens` を超えないよう切り詰め
6. LLM に渡して回答生成

**出力**:
```json
{
  "answer": "LLM の回答",
  "used_files": ["TAX001.md", "SUB002.md", ...],
  "total_tokens_sent": 187000,
  "retrieval_time_ms": 450,
  "llm_time_ms": 3200
}
```

この実装は、WS3 の成果物(`/impact-analysis`)とかなり重複するため、**WS3 のエンドポイントを再利用または拡張する形**が現実的。実装コストの最小化のため、段階 1 では「簡易版の全文送信」にとどめ、本格実装は採用確定後に行う。

## 7. 評価結果の使い道

### 7.1 採用判断フレーム

評価結果を以下 3 軸で判定する:

1. **精度**: 総合スコア 3.5 以上、致命的な的外れ回答が少ない
2. **速度**: 想定ユーザー数での応答時間が許容範囲(例: 1 質問 10 秒以内)
3. **運用コスト**: 構築・チューニング・保守が現実的(特に非 AI 専門者への引き継ぎ可能性)

### 7.2 想定される採用シナリオ

**シナリオ 1: 手法 A(既存 GraphRAG)が抜けて優秀**
→ 既存投資を活かす形で採用。自社ナレッジも既に構築できている
→ チャンクサイズ等のチューニングで追加改善を図る

**シナリオ 2: 手法 B(Dify 標準)が意外にも競合**
→ 保守性最優先で採用。構築コストが劇的に下がる
→ 複雑な質問への精度は犠牲にする判断

**シナリオ 3: 手法 C(全文送信)が全領域で高精度**
→ 新しいアプローチへの移行。WS3 のファイル関係グラフが本番運用に直結する
→ コンテキスト上限への対応(長大ファイルの扱い)を詰める必要がある

**シナリオ 4: 質問カテゴリで使い分け**
→ 「事実検索は B、影響範囲は C、概念説明は A」のように、Dify ワークフローで質問分類して振り分ける
→ 分岐ロジックは複雑化するが、精度は最大化される

**シナリオ 5: どれも決定的でない**
→ 運用コスト最優先で手法 B 暫定採用、本番運用中に改善を続ける
→ 段階 2 の結果で前処理方式だけ決めて、RAG は継続検討

### 7.3 採用後の作業

採用された手法に対して、以下の作業が発生する(本 WS2 の範囲外だが言及しておく):

- 本番運用に耐えるスケール検証(12 万ファイル投入テスト)
- ユーザー向け UI の作り込み(Dify ワークフローの画面設計)
- チャンキング戦略の詳細化(手法 A、B では特に必要)
- 運用マニュアルの整備(引き継ぎ先向け)
